# Device Lock / Unlock Protocol Spec — v1

Authoritative spec for the lock/unlock + location-tracking system.
Backend, Android app, and QA tests **must conform to this document.**

## 1. Roles

| Role | Component |
|---|---|
| **Server** | Express backend (`backend/`) |
| **Device** | Android Device-Owner app on customer phone |
| **FCM** | Firebase Cloud Messaging (free push channel) |
| **Operator** | Branch operator using the web UI |

## 2. Identifiers

- **`imei`** — primary device identifier (15 digits, captured at shop counter).
- **`device_token`** — FCM registration token (refreshed by Android occasionally).
- **`enrollment_token`** — one-time random 32-byte hex string, generated by the Server when an operator clicks "Enroll device". Embedded in the QR shown on screen and consumed by Android during provisioning. TTL 30 minutes.
- **`device_secret`** — 32-byte hex HMAC key, generated server-side at enrollment, stored encrypted on device. Used to sign every device→server request.

## 3. Enrollment (point-of-sale, 3–5 min)

```
Operator clicks "Enroll device" on the order page
   ↓
Server: insert into devices(...) with status='pending',
        generate enrollment_token + device_secret,
        return both
   ↓
UI shows QR code containing:
   { "v":1, "url":"https://<backend>/api/devices/enroll",
     "token":"<enrollment_token>", "secret":"<device_secret>",
     "branch":"<branch_id>", "order":"<order_id>" }
   ↓
At customer phone (factory-reset / new):
   1. Skip Google setup, six-tap "Welcome" screen → camera opens for QR
   2. Android Provisioning installs APK and grants Device Owner privilege
   3. App reads payload, calls POST /api/devices/enroll with:
        { token, imei, fcm_token, device_model, android_version }
   ↓
Server: validates token, marks device active, stores imei+fcm_token,
        returns ok + device_id
```

After enrollment, the device's `enrollment_token` is single-use and burned.

## 4. Lock / Unlock command flow

Operator clicks **Lock** on the order page → Server resolves the order's `device_imei` → looks up the active device → publishes an FCM message:

```json
{
  "to": "<fcm_token>",
  "priority": "high",
  "data": {
    "type": "command",
    "action": "lock" | "unlock",
    "command_id": "<uuid>",
    "issued_at": "2026-05-24T12:00:00Z",
    "reason": "Installment overdue 7 days",
    "lock_message": "Pay your installment - call 0300-1234567",
    "hmac": "<hex>"
  }
}
```

`hmac` = `HMAC-SHA256(device_secret, command_id + action + issued_at)`. The Android app **rejects any command with a bad HMAC** — this prevents a stolen FCM key from bricking phones.

Device responds with an **acknowledgement heartbeat** within 10 s. If no ack within 60 s, server marks the command `pending_delivery` and retries every 10 min for 24 h; after that it goes to `failed_delivery` and surfaces a red banner in the UI.

### Lock command outcome on device
- Foreground a full-screen Activity with `FLAG_SHOW_WHEN_LOCKED | FLAG_KEEP_SCREEN_ON | FLAG_DISMISS_KEYGUARD`
- Set the activity as the default Home launcher (via Device Owner policy) so Home button does nothing
- Block recents (`setLockTaskFeatures(LOCK_TASK_FEATURE_NONE)`)
- Block status bar pull-down (`setStatusBarDisabled(true)`)
- Disable app uninstall + Google account add (`addUserRestriction`)
- Persist `lock_state=locked` in encrypted SharedPreferences so a reboot reapplies

### Unlock command outcome
- Finish the lock Activity, restore default Home, clear restrictions
- Persist `lock_state=unlocked`

## 5. Heartbeat (device → server)

Every 30 min (configurable per branch), and immediately after enrollment, command-ack, SIM change, or boot:

```http
POST /api/devices/heartbeat
Authorization: HMAC <device_id>:<sig>
X-Issued-At: <iso8601>
Content-Type: application/json

{
  "imei": "...",
  "lock_state": "locked" | "unlocked",
  "last_command_id": "<uuid or null>",
  "last_command_status": "applied" | "failed",
  "battery_pct": 78,
  "network_type": "wifi" | "mobile" | "none",
  "sim_serial": "<iccid>",
  "lat": 32.4945,
  "lon": 74.5229,
  "accuracy_m": 12,
  "fcm_token": "<current>"
}
```

`sig` = `HMAC-SHA256(device_secret, body + X-Issued-At)`. Server rejects if signature invalid or `X-Issued-At` is more than 10 minutes off server time (replay defence).

Server upserts:
- `devices.last_seen_at`, `last_battery`, `last_network`, `current_sim_serial`, `fcm_token`
- New row in `device_locations` (only if `lat/lon` present)
- If `sim_serial` changed since last heartbeat → emit `sim_change` activity log + WhatsApp alert to branch

## 7. Server endpoints (final list)

| Method + Path | Role | Purpose |
|---|---|---|
| `POST /api/devices/enrollment-tokens` | admin/op | Mint a new enrollment_token + device_secret, return QR payload |
| `POST /api/devices/enroll` | (unauthenticated, token-gated) | Phone consumes the enrollment_token, registers itself |
| `POST /api/devices/heartbeat` | HMAC-authed device | Periodic check-in: lock state, location, SIM, battery |
| `POST /api/orders/:id/lock` | admin/op | Issue lock command (server picks the order's device, signs, FCM-pushes) |
| `POST /api/orders/:id/unlock` | admin/op | Issue unlock command |
| `GET /api/devices/:imei/locations?from=&to=` | admin/op | Location history for map view |
| `GET /api/devices/:imei/status` | admin/op | Last-seen + lock state + battery + SIM |
| `POST /api/devices/run-auto-lock` | admin (or cron) | Sweep overdue installments and auto-lock per branch policy |

## 8. Auto-lock policy

Configurable per branch in a new `branches.auto_lock_days` column (default `null` = manual only). If set to e.g. `7`, the nightly cron locks any device whose most recent overdue installment is ≥ 7 days past due. A WhatsApp warning is sent **24 h before** the auto-lock fires.

## 9. Auto-unlock

When `POST /installments/:id/pay` clears the overdue balance on an order whose `device_locked = true`, the server immediately issues an unlock command in the background. Operators see "auto-unlock issued" in the order timeline.

## 10. Security model

- **Device → server**: every request HMAC-signed with `device_secret`; server validates + checks timestamp ≤10 min skew (replay window). Compromised FCM token alone cannot impersonate a device.
- **Server → device**: every command HMAC-signed with the **same `device_secret`**; device rejects bad HMAC. Compromised FCM API key alone cannot brick devices.
- **Enrollment token**: single-use, 30 min TTL, only valid if the IMEI presented matches the device that scanned the QR.
- **All device endpoints rate-limited** at 60 req/min per device_id.

## 11. Failure modes

| Scenario | Behavior |
|---|---|
| Phone offline when lock issued | Command queued at FCM (4 weeks max); applied on next reconnect |
| Phone offline > 14 days | UI shows red "offline" indicator, suggests calling customer |
| FCM service down | Server retries every 10 min; queue persists in `device_commands` |
| Customer factory resets | App is Device Owner — reinstalls automatically from `disk_encryption_image_persistent` storage, re-enrolls with stored secret, restores lock state |
| Customer flashes custom firmware | Device disappears from system — flagged as "lost contact >14d", goes to legal recovery |
| Customer swaps SIM | Heartbeat detects, server emits `sim_change`, branch gets WhatsApp alert |

## 12. Frontend UI additions

OrderDetail page gets a **Device** card with:
- IMEI (copy button)
- Online status (green dot / red dot + last-seen relative time)
- Lock status badge (Locked / Unlocked / Pending Command)
- **Lock** + **Unlock** buttons (admin/operator)
- "Locate now" button (forces a heartbeat by sending a no-op command)
- Last known location (mini Leaflet/MapLibre map, no API key needed)
- "View full location history" link → modal with map of last 30 days

New top-level **Devices** page (admin): table of all enrolled devices with filter by branch, last-seen, lock status. Click a row → device detail + map.

## 13. Out of scope for v1

- iPhone (no enforcement path)
- Two-way customer chat from lock screen
- Photo capture from device camera
- Microphone activation
